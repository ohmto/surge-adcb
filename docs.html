Automated Tasks

Font Generator

This task can be used from the command line to generate font from the SVG icons found in the src/assets/toolkit/font-icons directory. This task is not part of the watch command to avoid re-generating fonts on each document save. Below command can be used to generate the font.

gulp iconfont

Brand Color Stylesheet Generator

This task can be used from the command line to generate brand colors from /src/data/toolkit.yml YAML file. The brand color stylesheet will be generated in /src/assets/toolkit/styles/settings/_brand-colors.scss. This task is not part of the watch command to avoid re-generating fonts on each document save. Below command can be used to generate the color stylesheet along with the right SASS variables.

gulp styles:brand

CSS Methodology

For CSS Architecture SMACSS would be used guidelines. Detailed guidelines on writing modular styles can be found at SMACSS Guidelines

Starting with BEM

For naming convention we would be using BEM (Block Element Modifier). Detailed guidelines on BEM can be found at BEM Guidelines. CSS Naming section provides detailed conventions used.

Linting

For SASS or CSS linting Stylelint is being used with gulp-stylelint to hook gulp watch and start tasks. The paths for files to be linted are defined in gulp.config.js file under toolkit object as below:

toolkit: {
  src: 'src/assets/toolkit/styles/toolkit.scss',
  brandStyles: {'src' : 'src/data/toolkit.yml', 'dest' :"src/assets/toolkit/styles/settings/_brand-colors.scss"},
  lint:
    [
      //get all .scss files in the src/assets/toolkit/styles folder.
      'src/assets/toolkit/styles/**/*.scss',
      //paths starting with ! will be ignored.
      '!src/assets/toolkit/styles/_font-icons.scss',
      '!src/assets/toolkit/styles/template/_font-icons.scss'
    ],
    dest: 'dist/assets/toolkit/styles',
    watch: 'src/assets/toolkit/styles/**/*.scss'
}

Css Naming

Namespacing

In no particular order, here are the individual namespaces and a brief description. We’ll look at each in more detail in a moment, but the following list should acquaint you with the kinds of thing we’re hoping to achieve.

  • o-: Signify that something is a global object and may be used in any number of unrelated contexts to the one you can currently see it in. Making modifications to these types of class could potentially have knock-on effects in a lot of other unrelated places. Tread carefully.
  • c-: Signify that something is a Component. This is a concrete, implementation-specific piece of UI. All of the changes you make to its styles should be detectable in the context you’re currently looking at. Modifying these styles should be safe and have no side effects.
  • u-: Signify that this class is a Utility class. It has a very specific role (often providing only one declaration) and should not be bound onto or changed. It can be reused and is not tied to any specific piece of UI.
  • t-: Signify that a class is responsible for adding a Theme to a view. It lets us know that UI Components’ current cosmetic appearance may be due to the presence of a theme.
  • is-, has-: Signify that the piece of UI in question is currently styled a certain way because of a state or condition. This stateful namespace is gorgeous, and comes from SMACSS. It tells us that the DOM currently has a temporary, optional, or short-lived style applied to it due to a certain state being invoked.
  • js-: JavaScript namespaces are pretty common now, and most people tend to use them. The idea is that—in order to properly separate our concerns—we should never have styling and behaviour bound to the same hooks. To bind both technologies onto the same hook means we can’t have one without the other: our UI becomes all-or-nothing, which makes it very opinionated and inflexible.

Our Approach (BEVM)

We follow a little bit modified version of BEM which solves some of the pains in BEM. BEVM (block__element--variation -modifier).

Variations (--variation)

We would like to avoid variations as much as possible. A variation would be defined as a different or distinct form or version of a block where we change a block on multiple levels in terms of size, color & layout etc. in the same class. Its mostly referred as Single Class approach in more classical BEM. For example we have a button block as below:

<button class="o-btn" />

To create a variation of the 'btn' block we created a class 'btn--primary' as below:

<button class="o-btn--primary" />

We do not repeating the base class like btn btn--primary and SASS we use @extend to share rules between the base and the variation. For example as below:

.o-btn {
  background: $brand-alto-gray;
  border: 0;
  border-radius: 2px;
  font-size: 1.3rem;
  font-weight: 500;
  padding: 12px 24px;
}

.o-btn--primary {
  @extend .btn;
  background: $brand-primary-light;
  background: $brand-gradient-red;
  color: $white;  
}

Variation Rules

  1. Can only apply one "variation" at a time
  2. No need to repeat base class name
  3. Can use SASS @extend to share rules
  4. Or a custom variation can be used without using @extend

Modifier (-modifier)

A modifier is an extension that makes a partial or minor change in the rule set of a block. So instead of using the classical modifier style where repeated btn creating redundancy as below:

<!-- not to be used -->
<button class="o-btn o-btn--warning o-btn--large o-btn--round">

We will be using chaining modifiers. Where modifiers are denoted by a leading (-) hyphen. So our HTML would be become:

<button class="o-btn--warning -size-large -border-round">

Our SASS structure would become:

.o-btn {
  &.-large {...}
  &.-round {...}
  &.-icon {...}
}
.o-btn--warning {
  @extend .o-btn;
  ...
}

Chainable Modifiers Rules

  1. Chainable modifiers should never modify the same property twice e.g. 2 more more chained modifiers change the color rule should not happen
  2. Use a namespace which describe the change e.g. size, color & width etc
  3. Use generic descriptors e.g. large, primary & dark etc

Customization & Bundling

Customization

Generate theme

Execute gulp createTheme --name <theme-name>

To add a new theme, perform this step. It generates all the necessary files and edits to the folder structure and files for the theme - <theme-name>. Below mentioned are the files and folders necessary to get the new theme off the ground. Example Usage:

gulp createTheme --name Private

What goes where

1. Under src/assets/toolkit/scripts/<theme-name> folder

Add theme-specific scripts here

2. Under src/assets/toolkit/styles/<theme-name> folder

Add theme-specific styles and style overrides here

3. Under src/data/<order-number>-<theme-name>/ folder

  1. toolkit.yml file - is where you will store theme(brand)-specific colors. Adding the file with colors here, adds a new section on the overview page with the mentioned colors. The <theme-name> will be the name of the section under which the colors appear. The <order-number> helps organize more priority brands above lesser priority brands. Lower the order number, higher the priority. If priority is not specified in the folder name, the sorting will resort to default order. It is advisable to keep the core folder as 00-core, to prioritize it over other folders.

  2. Running gulp brandstyles will generate theme specific .scss files. Note: the sass value of colors must be matched across all toolkit files as the colors are mapped in the scss. Missing colors will cause issues in toolkit build. Colors that are site-agnostic should be stored in core/toolkit.yml and can be accessed across all themes.

  3. The .json files you save in this folder will be used as content at a component level in this theme's pages only. If a relevant .json file doesn't exist in <order-number>-<theme-name>/ folder data will be populated from the 00-core folder.

4. Under src/materials/components/components-<theme-name> folder

Add all the theme specific components.

5. Under src/views/pages/<theme-name> folder

Add all the theme pages here.

Points to Remember

Colors

  1. Colors that will be applied across the site need to be added in data/core/toolkit.yml file. Add colors under appropriate category, prefix based on the category the colors fall in.
  2. Colors that will be applied at the theme level need to be added in data/<theme-name>/toolkit.yml file. They should be prefixed with $brand-.
  3. In .yml files, it is good practice to add the color's human-friendly reference name, above the hex and sass value. This can help users identify the color in conversations. For example,
    Generic Personal Red:
      hex: "#ec0100"
      sass: $generic-red

where, Generic Personal Red in the block above is a human-friendly reference name

Cross Theming

What is cross-theming?

Fabricator develops themes and references colors based on the theme folders. Let's say there are three themes

  • Light (background - white, font - blue)
  • Medium (background - grey, font - green) and
  • Dark (background - black, font - yellow)

However, we may be required to add a component with theme Dark on a page with theme Medium. The following steps help resolve this scenario.

Steps

  1. Set themes colors in the toolkit.yml as we normally do.

  2. Run gulp brandstyles this will generate _brand-colors.scss and also creates _src/assets/toolkit/styles/core/common/site-themes.scss

  3. In the actual style file (*.scss) -

.myclass {
  font-size: 15px;
  ...other-css-properties
  color: $brand-primary-font;                       -- Line 1
  background-color: $brand-bg-color;                -- Line 2
  @include themify($themes) {
    color: themed('brand-primary-font');            -- Line 3
    background-color: themed('brand-bg-color');     -- Line 4
  }
}

In the above example to add theming functionality on Line 3 & 4, we have added code in the below format.

@include themify($themes) {
  <css-property>: themed (<property-name-in-toolkit.yml-file>)
}

  1. Also note, we need to keep the <css-property> in the same manner as in Line 1 & 2.

  2. On the component on which you wish to apply the theme, add class t-<theme-name>. For instance, to apply the Dark theme on the Medium page, add class t-dark for the required component.

Conventions

  1. Keep the themify color at the bottom of the style block.

  2. The property in the theme block needs to be placed right above themify to help in quick referencing and updating the same.

Points to Remember

The theme will be applied on all child elements not on the element itself.

Fabricator Developer

Behind the scenes - gulp createTheme

Let's have a look at what's happening, when you run the gulp createTheme command:

Assuming a theme private needs to be created. We will run the command - gulp createTheme --name Private

  1. Under src/assets/toolkit/scripts/, the folder private is generated. Within this folder, toolkit.js is created.

  2. Under src/assets/toolkit/styles/, the folder private is generated. Within this folder, toolkit.scss is created.

  3. Under src/data/, the folder <order-number>-private is generated. Within this folder, toolkit.yml is created.

  4. Under src/materials/components/ folder, the folder components-private is generated.

  5. Under src/meta/config.json an entry is added.

    {
  "displayName": "Private",
  "folderName": "private",
  "landingPageName": "private"
}

  6. Under src/views/ folder, 2 new files are added.

    1. private.html and
    2. components-private.html.

  7. Under src/views/layouts/includes folder, the file f-menu-private.html is added, containing the menu for the new theme.

  8. Under the same folder, the new theme's f-menu-list menu file is added as reference.

  9. Under src/views/pages folder, a new folder private is added. Within this folder, are added

    1. private-arabic and
    2. private-english.

  10. Under gulp-config folder, a new entry is made in

    1. scripts.json - path to the toolkit.js file, required by the gulp.config file,
    2. styles.json - path to the toolkit.scss file, required by the gulp.config file,
    3. webpack-script-entry.json - path to the toolkit.js file, required by the webpack.config.js file to generate toolkit.js files in dist.

Note: To get the new branch working properly, you will have to populate the necessary brand colors values in toolkit.yml.

In addition, you will also need to run gulp brandstyles to generate the brand's brand_colors.scss file.
